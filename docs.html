🔭 What ConvOps Monitors

ConvOps connects to your AWS account via a read-only IAM role and monitors CloudWatch Alarms in real time — across CPU, memory, error rate, latency, and any custom metric you've configured.

Alarm Types

💻 CPU Utilization

Threshold alarms on EC2, ECS tasks, RDS, Lambda concurrency.

🧠 Memory Usage

Container-level and instance-level memory alarms.

🚨 Error Rate

5xx rates, Lambda errors, application error metrics.

⏱️ Latency

API Gateway response times, Lambda duration, ALB target response time.

🔧 Custom Metrics

Any CloudWatch alarm you've created — ConvOps handles it automatically.

Supported AWS Resources

🐳 ECS 🖥️ EC2 🗄️ RDS ⚡ Lambda ⚖️ ALB 🌐 API Gateway 🏎️ ElastiCache 📨 SQS 📦 DynamoDB 🪣 S3

Alert Channels

💬 WhatsApp # Slack

Choose one or both. Alerts land wherever your team already is — no new tools.

Alert Conditions

🔴
ALARM state → Immediate notification
When a CloudWatch alarm transitions to ALARM, ConvOps ingests it instantly, runs AI analysis, and delivers a rich notification with root cause — not just a raw metric value.
✅
OK state → Resolved notification
When the alarm returns to OK, you receive a resolved message so you know the incident closed — no need to manually check CloudWatch.
🔁
Repeat-alert deduplication
If the same alarm fires repeatedly (flapping), ConvOps batches repeat notifications and only pages you again after repeat_alert_threshold fires (default: 5). Prevents alert fatigue from noisy alarms. Configurable per workspace.

🔍 Investigation Capabilities

Reply 4 to any alert to trigger a deep investigation. ConvOps pulls data from multiple AWS sources simultaneously, correlates everything, and delivers an AI root cause analysis — within seconds, without opening AWS console.

💡 The first alert already includes AI analysis. Reply "4" when you want the full deep-dive with all data sources — metrics, logs, CloudTrail, cost, security groups, and more.
📊
CloudWatch Metrics — last 30 min
Avg + max values for the relevant metric. Shows the trend leading up to the alarm threshold breach so you can see if it's a spike or a gradual increase.
📋
Application Logs — last 15 min
Scans CloudWatch Logs filtered for: ERROR WARN Exception timeout OOM. Surfaces the most relevant log lines without you having to search.
🐳
ECS Service State
Running / desired / pending task counts. Deployment status. Failed task details. Stopped task reasons + exit codes — the exact message that tells you why a container stopped.
🖥️
EC2 Instance State
Instance state, instance type, CPU credit mode (for burstable T-series). Confirms whether the instance is running or in a degraded/impaired state.
🗄️
RDS Database Status
DB status, engine version, connectivity analysis including security group rules and VPC subnet configuration. Identifies whether the database is reachable from your application layer.
🛡️
Security Group Audit
Inbound rules check. Missing port detection — flags if a required port (e.g., 5432 for PostgreSQL, 3306 for MySQL, 6379 for Redis) is not open to expected sources.
🌐
API Gateway Diagnostics
5xx and 4xx error rates. CORS configuration status. Stage info (deployment ID, throttling limits). Distinguishes client errors from server-side issues.
⚖️
ALB Target Health
Per-target group breakdown: healthy / unhealthy / draining target counts. Identifies which backend instances have dropped out of rotation.
💰
Cost Analysis — last 7 days
Per-service spend breakdown. Spend spike detection: flags any service with cost >20% above its 7-day rolling average. Useful for spotting runaway resources during incidents.
📜
CloudTrail — last 5 changes (past 2h)
The last 5 changes to the affected resource in the past 2 hours. Shows who changed what — deploys, config changes, SG modifications, IAM updates. Correlates changes to incidents.
🤖
Claude AI — Root Cause Analysis
All collected data is fed to Claude AI, which produces 3–5 bullet root cause analysis + recommended actions. Correlates metrics, logs, config changes, and resource state into a single coherent diagnosis with suggested next steps.

🏗️ Customer Stack — What Gets Deployed

ConvOps deploys a single CloudFormation stack into your AWS account. Here is exactly what it creates — no surprises.

🔗 One-click setup. Go to app.convops.io, enter your AWS Account ID, and click Launch Stack. CloudFormation handles everything below automatically.

🔑 IAM Role: ConvOpsAccessRole

  • Trusted by: ConvOps AWS account 009001720832 — only ConvOps can assume this role.
  • ExternalId: A unique token generated per customer, stored in Secrets Manager. Prevents confused-deputy attacks — even if someone knows your role ARN, they cannot assume it without the ExternalId.
  • Managed policy attached: arn:aws:iam::aws:policy/ReadOnlyAccess — AWS-managed read-only access to all services. ConvOps can read metrics, logs, and resource state but cannot modify anything.

Additionally, a separate inline policy ConvOpsRemediationPolicy grants the following write actions — only executed when you explicitly confirm:

ecs:UpdateService
Restart or scale ECS services
ec2:RebootInstances
Reboot EC2 instances
rds:RebootDBInstance
Reboot RDS databases
lambda:InvokeFunction
Invoke Lambda functions

📣 SNS Topic: ConvOpsAlertTopic

  • Receives CloudWatch alarm state change notifications.
  • Subscribed to the ConvOps ingestion endpoint via HTTPS: https://ewvdzp6c79.execute-api.eu-central-1.amazonaws.com/prod/ingest
  • You attach this SNS topic as an action on your CloudWatch alarms — that's the only wiring needed.

📅 EventBridge Rule (optional)

  • Catches CloudWatch alarm state changes automatically, without needing SNS actions on each alarm individually.
  • Useful for accounts with many alarms — set-and-forget routing.

🔒 Security Model

ConvOps has cross-account access to your AWS environment. Here's exactly what that means — no vague claims.

🔑 No stored AWS credentials

ConvOps never stores your AWS credentials. Access works via STS AssumeRole — short-lived tokens with a 15-minute TTL. Tokens expire automatically.

🛡️ ExternalId protection

Your IAM role requires a unique ExternalId (generated per customer, stored in Secrets Manager). Prevents confused-deputy attacks — nobody else can assume your role.

👁️ Read-only by default

The ReadOnlyAccess managed policy covers all reads. Write actions (ecs:UpdateService, ec2:RebootInstances, rds:RebootDBInstance) require your explicit confirmation for every individual action.

📱 Phone binding

Only the person who received the alert on their registered number/Slack account can confirm actions. Confirmation expires after 5 minutes.

📋 Audit log — 90 days

Every investigation and action is logged: timestamp, resource ARN, who approved, outcome. Exportable on request. 90-day retention.

🗃️ Minimal data storage

We store only operational metadata: alarm events, conversation state. Never raw application logs, log content, secrets, or business data.

🇪🇺 EU data residency

All ConvOps infrastructure runs in eu-central-1 (Frankfurt). Data does not leave the EU. GDPR compliant — see convops.io/privacy.

🗑️ Revoke instantly

Delete the ConvOpsAccessRole from your AWS console. ConvOps loses all access immediately — no support ticket, no waiting period.

🎬 Actions Available

When you confirm an action (reply 1 + YES), ConvOps can execute the following remediation actions on your behalf. Every action requires two-step confirmation and is logged.

🔄 Restart ECS service
Forces a new deployment — stops and replaces all running tasks. Clears hung containers, OOM state, connection pool saturation.
📈 Scale up ECS (+1 task)
Increments desired count by 1. Absorbs traffic spikes without a full restart.
📉 Scale down ECS (-1 task)
Decrements desired count by 1. Useful for cost reduction or isolating a bad task.
⏪ Rollback ECS
Redeploys the previous task definition revision. Reverts a bad deploy without touching code.
🔌 Reboot EC2 instance
Issues a soft reboot via AWS API. Useful for clearing transient kernel/OS issues without data loss.
🗄️ Reboot RDS instance
Restarts the database engine. Clears connection pool saturation and recovers from transient DB failures.
⏳ Action execution is coming in the Pro plan. The current free plan includes alerts + full AI investigation. Actions (restart, scale, reboot) are in active development.

⚙️ Configurable Settings

Adjust ConvOps behaviour per workspace from the app dashboard.

🔁
repeat_alert_threshold
How many times the same alarm must fire before ConvOps sends a repeat notification. Default: 5. Set lower for critical alarms, higher for noisy ones. Configured per workspace.
📢
Alert channels
Choose WhatsApp, Slack, or both. You can route different alarm types to different channels (e.g., P1 to WhatsApp, P2/P3 to Slack).
📞
On-call routing
Route alerts to specific users. Configure who gets action buttons (can confirm remediation) vs who gets read-only notifications (informed only). Useful for on-call rotations.
