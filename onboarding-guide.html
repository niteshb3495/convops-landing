Customer Onboarding Guide

Everything you need to connect your AWS infrastructure to ConvOps. Takes about 5 minutes.

1

Deploy CloudFormation

IAM Role + SNS Topic. ~2 min.

2

Tag Your Resources

Mark what ConvOps can act on. ~1 min.

No Lambda functions. No code changes. No agents in your account.
ConvOps only needs an IAM role (to read and act) and an SNS topic (to receive alarm notifications).

What You'll Deploy

ResourceTypePurpose
ConvOpsAccessRole IAM Role Allows ConvOps to read your metrics and take approved actions
ConvOpsAlertTopic SNS Topic Receives CloudWatch alarms and forwards them to ConvOps

That's it. No changes to existing resources.

Step 1 — Deploy the CloudFormation Template

This creates the IAM Role and SNS Topic ConvOps needs. Replace the placeholder values with the credentials provided during onboarding.

Before deploying: Replace YOUR_CUSTOMER_EXTERNAL_ID and YOUR_CONVOPS_API_KEY with the values provided by ConvOps.

The Template

AWSTemplateFormatVersion: '2010-09-09'
Description: ConvOps Infrastructure Setup — IAM Role + Alert Topic

Parameters:
  CustomerExternalId:
    Type: String
    Description: Unique external ID provided by ConvOps (keep this secret)
    Default: YOUR_CUSTOMER_EXTERNAL_ID

  ConvOpsApiKey:
    Type: String
    Description: Your ConvOps API key (provided during onboarding)
    Default: YOUR_CONVOPS_API_KEY
    NoEcho: true

Resources:

  # SNS Topic: receives CloudWatch alarms → forwards to ConvOps
  ConvOpsAlertTopic:
    Type: AWS::SNS::Topic
    Properties:
      TopicName: ConvOpsAlerts
      Subscription:
        - Protocol: https
          Endpoint: https://api.convops.io/v1/ingest
          DeliveryPolicy:
            healthyRetryPolicy:
              numRetries: 3
              minDelayTarget: 20
              maxDelayTarget: 20

  ConvOpsAlertTopicPolicy:
    Type: AWS::SNS::TopicPolicy
    Properties:
      Topics:
        - !Ref ConvOpsAlertTopic
      PolicyDocument:
        Statement:
          - Effect: Allow
            Principal:
              Service: cloudwatch.amazonaws.com
            Action: sns:Publish
            Resource: !Ref ConvOpsAlertTopic

  # IAM Role: allows ConvOps to read and take approved actions
  ConvOpsAccessRole:
    Type: AWS::IAM::Role
    Properties:
      RoleName: ConvOpsAccessRole
      AssumeRolePolicyDocument:
        Version: '2012-10-17'
        Statement:
          - Effect: Allow
            Principal:
              AWS: arn:aws:iam::009001720832:role/convops-executor
            Action: sts:AssumeRole
            Condition:
              StringEquals:
                sts:ExternalId: !Ref CustomerExternalId
      Policies:
        - PolicyName: ConvOpsReadPolicy
          PolicyDocument:
            Version: '2012-10-17'
            Statement:
              - Effect: Allow
                Action:
                  - cloudwatch:GetMetricData
                  - cloudwatch:GetMetricStatistics
                  - cloudwatch:DescribeAlarms
                  - logs:FilterLogEvents
                  - logs:GetLogEvents
                  - logs:DescribeLogGroups
                  - logs:DescribeLogStreams
                  - ec2:DescribeInstances
                  - ecs:DescribeTasks
                  - ecs:DescribeServices
                  - ecs:ListTasks
                  - ecs:ListServices
                  - rds:DescribeDBInstances
                  - rds:DescribeEvents
                  - lambda:ListFunctions
                  - lambda:GetFunctionConfiguration
                Resource: "*"

        - PolicyName: ConvOpsActionPolicy
          PolicyDocument:
            Version: '2012-10-17'
            Statement:
              - Effect: Allow
                Action: ecs:UpdateService
                Resource: "*"
                Condition:
                  StringEquals:
                    aws:ResourceTag/ConvOpsManaged: "true"
              - Effect: Allow
                Action: ec2:RebootInstances
                Resource: "*"
                Condition:
                  StringEquals:
                    aws:ResourceTag/ConvOpsManaged: "true"
              - Effect: Allow
                Action: rds:RebootDBInstance
                Resource: "*"
                Condition:
                  StringEquals:
                    aws:ResourceTag/ConvOpsManaged: "true"
              - Effect: Allow
                Action: ssm:SendCommand
                Resource:
                  - !Sub "arn:aws:ssm:*:*:document/AWS-RestartService"
                  - !Sub "arn:aws:ssm:*:*:document/AWS-RunShellScript"
                  - !Sub "arn:aws:ec2:*:${AWS::AccountId}:instance/*"
                Condition:
                  StringEquals:
                    aws:ResourceTag/ConvOpsManaged: "true"

Outputs:
  RoleArn:
    Description: Share this with ConvOps during onboarding
    Value: !GetAtt ConvOpsAccessRole.Arn
  SNSTopicArn:
    Description: Use this as your CloudWatch alarm action
    Value: !Ref ConvOpsAlertTopic

Deploy via AWS CLI

aws cloudformation deploy \
  --template-file customer-convops-setup.yaml \
  --stack-name convops-setup \
  --capabilities CAPABILITY_NAMED_IAM \
  --parameter-overrides \
    CustomerExternalId=YOUR_EXTERNAL_ID \
    ConvOpsApiKey=YOUR_API_KEY

Or via AWS Console

CloudFormation → Create Stack → Upload template file → Follow wizard.

After deployment, note the RoleArn and SNSTopicArn from the stack Outputs tab. You'll need them in Steps 3 and 4.

Step 2 — Tag Resources You Want ConvOps to Manage

ConvOps will only take actions on resources you explicitly tag. Investigation (reading metrics and logs) works on all resources — actions only on tagged ones.

Tag KeyTag Value
ConvOpsManagedtrue

Tag an ECS service

aws ecs tag-resource \
  --resource-arn arn:aws:ecs:eu-central-1:YOUR_ACCOUNT:service/my-cluster/my-service \
  --tags key=ConvOpsManaged,value=true

Tag an EC2 instance

aws ec2 create-tags \
  --resources i-0abc123def456 \
  --tags Key=ConvOpsManaged,Value=true

Tag an RDS instance

aws rds add-tags-to-resource \
  --resource-name arn:aws:rds:eu-central-1:YOUR_ACCOUNT:db:my-db \
  --tags Key=ConvOpsManaged,Value=true
Resources without this tag: ConvOps can investigate (read logs/metrics) but cannot take any action. You stay in full control of what's actionable.

Step 3 — Connect Your CloudWatch Alarms

Add the ConvOpsAlerts SNS topic as an alarm action on any alarm you want ConvOps to monitor.

Via AWS CLI

# Get your SNS topic ARN from CloudFormation outputs
SNS_ARN=$(aws cloudformation describe-stacks \
  --stack-name convops-setup \
  --query "Stacks[0].Outputs[?OutputKey=='SNSTopicArn'].OutputValue" \
  --output text)

# Add to an existing alarm
aws cloudwatch put-metric-alarm \
  --alarm-name "prod-api-cpu-high" \
  --alarm-actions $SNS_ARN \
  --metric-name CPUUtilization \
  --namespace AWS/ECS \
  --statistic Average \
  --period 300 \
  --threshold 85 \
  --comparison-operator GreaterThanThreshold \
  --evaluation-periods 2

Via AWS Console

CloudWatch → Alarms → select your alarm → Edit → Notification section → Add action → SNS topic → select ConvOpsAlerts → Save.

Confirm the SNS subscription: After deployment, check SNS → Subscriptions in the console. The HTTPS subscription to ConvOps should show Confirmed. If it shows Pending, contact nitz@convops.io.

Step 4 — Complete Onboarding on ConvOps

  1. Go to convops.io → register your account
  2. Paste the IAM Role ARN from CloudFormation stack outputs
  3. Add your WhatsApp number to receive alerts
  4. Test: manually trigger a CloudWatch alarm → you should receive a WhatsApp message within 30 seconds
Having trouble? Email nitz@convops.io with your CloudFormation stack name and we'll help you debug within a few hours.

What ConvOps Can and Cannot Do

✅ ConvOps CAN

Read CloudWatch metrics and alarms

Read CloudWatch Logs

Describe EC2, ECS, RDS, Lambda resources

Restart ECS services (tagged only)

Reboot EC2 instances (tagged only)

Reboot RDS instances (tagged only)

Run specific SSM documents (tagged only)

❌ ConvOps CANNOT

Terminate or delete any resource

Modify security groups, VPCs, or IAM policies

Access S3, Secrets Manager, or any secrets

Act without your explicit WhatsApp confirmation

Act on resources without ConvOpsManaged: true

Security Notes

ExternalId
Your unique ExternalId prevents other AWS accounts from assuming your ConvOps role. Treat it like a password — keep it secret.
Revoke access instantly
Delete the ConvOpsAccessRole IAM role at any time. ConvOps immediately and permanently loses all access.
Confirmation gate
Every action requires an explicit "YES" reply on WhatsApp before execution. Nothing happens automatically.
Data handling
ConvOps never stores your logs or metrics. Fetched on-demand, used, discarded. Operational metadata (alarm names, action history) retained 90 days.
Audit trail
All actions logged with timestamp, phone number, and outcome. Available on request.
Session tokens only
No long-lived credentials stored. ConvOps uses STS session tokens (15-min TTL) for all cross-account operations.

Troubleshooting

Not receiving alerts?

Check that the SNS subscription shows Confirmed (not Pending) in the SNS console. Also verify your CloudWatch alarm has the SNS topic set as the alarm action — not just the OK action.

Action failed with permission denied?

Verify the target resource has the ConvOpsManaged: true tag. IAM actions are blocked on untagged resources by design.

CloudFormation deploy failed?

Ensure your IAM user/role has cloudformation:*, iam:CreateRole, iam:PutRolePolicy, and sns:CreateTopic permissions. Add the --capabilities CAPABILITY_NAMED_IAM flag to the CLI command.

SNS subscription stuck on Pending?

ConvOps needs to confirm the HTTPS subscription from our side. Email nitz@convops.io with your SNS Topic ARN and we'll confirm it manually within a few hours.
